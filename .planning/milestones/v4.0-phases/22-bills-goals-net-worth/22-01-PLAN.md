  <files>

  <action>
    **Migration SQL** (`supabase/migrations/20260223000002_create_bills_goals_net_worth.sql`):
    Create all 5 tables using the schemas from 22-RESEARCH.md:
    - `recurring_bills` with household_id FK, plaid_stream_id, name, amount_cents (BIGINT), frequency CHECK, user_status CHECK (auto/confirmed/dismissed), source CHECK (plaid/manual), previous_amount_cents for price change detection, UNIQUE(household_id, plaid_stream_id)
    - `savings_goals` with household_id FK, target_cents (BIGINT), current_cents (BIGINT), deadline (DATE optional), funding_type CHECK (manual/linked), linked_account_id FK to accounts, icon, color, status CHECK (active/completed/archived)
    - `goal_contributions` with goal_id FK CASCADE, amount_cents (BIGINT), note, contributed_at
    - `net_worth_snapshots` with household_id FK, snapshot_date (DATE), total_cents/assets_cents/liabilities_cents (BIGINT), UNIQUE(household_id, snapshot_date)
    - `manual_assets` with household_id FK, name, value_cents (BIGINT), asset_type CHECK (property/vehicle/investment/other), notes

    For each table:
    - Add IN-subquery RLS policies (same pattern as budgets table: `household_id IN (SELECT household_id FROM household_members WHERE user_id = (SELECT auth.uid()))`)
    - For goal_contributions: RLS via join through savings_goals.household_id
    - Add household_id and relevant secondary indexes
    - Add `updated_at` trigger for tables that need it (recurring_bills, savings_goals, manual_assets)

    **RecurringBillsDB** (`lib/db/recurring-bills.ts`):
    - Constructor takes SupabaseClient (same pattern as BudgetsDB)
    - `getByHousehold(householdId)`: Get all bills, ordered by next_due_date. Do NOT filter by user_status -- return all bills so UI can group them (active vs dismissed sections).
    - `create(bill)`: Insert a new bill (for manual creation)
    - `update(id, updates)`: Update bill fields (name, amount_cents, frequency, next_due_date, user_status, previous_amount_cents)
    - `upsertFromPlaid(householdId, bills: DetectedBill[])`: Upsert detected bills by plaid_stream_id. For existing bills, update amount/frequency/predicted_next_date/is_active/plaid_status. If amount changed, set previous_amount_cents to old amount. For new bills, insert with user_status='auto'.
    - `delete(id)`: Delete a bill

    **SavingsGoalsDB** (`lib/db/savings-goals.ts`):
    - Constructor takes SupabaseClient
    - `getByHousehold(householdId)`: Get all active goals with contribution summary (total contributed, last contribution date). For linked goals, current_cents reflects the linked account balance.
    - `create(goal)`: Insert new goal
    - `update(id, updates)`: Update goal fields
    - `delete(id)`: Delete goal and cascade contributions
    - `addContribution(goalId, amountCents, note?)`: Insert contribution AND update goal.current_cents (add amount)
    - `getContributions(goalId)`: Get contribution history, ordered by contributed_at DESC

    **NetWorthSnapshotsDB** (`lib/db/net-worth-snapshots.ts`):
    - Constructor takes SupabaseClient
    - `upsert(householdId, date, totalCents, assetsCents, liabilitiesCents)`: Upsert snapshot (uses ON CONFLICT on household_id+snapshot_date unique constraint). Use `.upsert()` Supabase method.
    - `getHistory(householdId, fromDate, toDate)`: Get snapshots in date range, ordered by snapshot_date ASC for chart
    - `getLatest(householdId)`: Get most recent snapshot

    **ManualAssetsDB** (`lib/db/manual-assets.ts`):
    - Constructor takes SupabaseClient
    - `getByHousehold(householdId)`: Get all manual assets
    - `create(asset)`: Insert new asset
    - `update(id, updates)`: Update asset
    - `delete(id)`: Delete asset

    **Types** (`lib/db/types.ts`):
    Add TypeScript interfaces for: RecurringBill, SavingsGoal, GoalContribution, NetWorthSnapshot, ManualAsset. Follow existing pattern (e.g., Budget interface). All money amounts as `number` (representing cents).

    **Barrel exports** (`lib/db/index.ts`):
    Add exports for RecurringBillsDB, SavingsGoalsDB, NetWorthSnapshotsDB, ManualAssetsDB.

  <action>
    **Plaid recurring fetch** (`lib/plaid/recurring.ts`):
    Create `fetchRecurringTransactions(accessToken: string)` function following the pattern from 22-RESEARCH.md:
    - Call `plaidClient.transactionsRecurringGet({ access_token: accessToken })`
    - Transform `outflow_streams` and `inflow_streams` into `DetectedBill[]` array
    - Apply sign convention: `toCents(-(stream.last_amount?.amount ?? 0))` to invert Plaid's positive-outflow to our negative-outflow convention
    - Extract: plaid_stream_id (stream.stream_id), account_id, merchant_name, description, amount_cents, frequency, predicted_next_date, first_date, last_date, is_active, status (MATURE/EARLY_DETECTION/TOMBSTONED), category_primary from personal_finance_category
    - Export `DetectedBill` interface
    - Import createPlaidClient from `./client`
    - Import toCents from `@/lib/money/arithmetic`

    **Bills validation** (`lib/validations/bills.ts`):
    - `billCreateSchema`: name (string, min 1, max 100), amount (number, positive -- UI handles sign, API converts to negative cents), frequency (enum: WEEKLY, BIWEEKLY, SEMI_MONTHLY, MONTHLY, ANNUALLY), next_due_date (string, ISO date, optional)
    - `billUpdateSchema`: Partial of create + user_status (enum: auto, confirmed, dismissed)
    - `billSyncSchema`: For triggering Plaid recurring sync (no body needed, just validates presence)
    - Export inferred types: BillCreateValues, BillUpdateValues

    **Goals validation** (`lib/validations/goals.ts`):
    - `goalCreateSchema`: name (string, min 1, max 100), target_amount (number, positive -- dollars, converted to cents at API), deadline (string, ISO date, optional), funding_type (enum: manual, linked), linked_account_id (string UUID, optional -- required when funding_type is linked, use .refine()), icon (string, optional), color (string, optional)
    - `goalUpdateSchema`: Partial of create + status (enum: active, completed, archived)
    - `contributionCreateSchema`: amount (number, positive -- dollars), note (string, optional, max 200)
    - `manualAssetCreateSchema`: name (string, min 1, max 100), value (number, positive -- dollars), asset_type (enum: property, vehicle, investment, other), notes (string, optional, max 500)
    - `manualAssetUpdateSchema`: Partial of create
    - Export inferred types

    Run `pnpm lint` and `pnpm exec tsc --noEmit` to verify.
  </action>
