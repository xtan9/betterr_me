**1. Add two new methods to `JournalEntriesDB` class in `lib/db/journal-entries.ts`:**

a) `getRecentEntryDates(userId: string, beforeDate: string, limit = 400): Promise<string[]>` — lightweight query selecting only `entry_date` from `journal_entries` where `user_id = userId` and `entry_date <= beforeDate`, ordered DESC, limited. Returns array of date strings. This powers streak computation without fetching full entry content.

b) `getEntriesForDates(userId: string, dates: string[]): Promise<JournalEntry[]>` — fetch entries matching specific dates using `.in("entry_date", dates)`. Select `id, entry_date, mood, title, content, word_count`. Returns matching entries ordered by `entry_date` DESC. This powers On This Day.

**2. Create `lib/journal/streak.ts` with two pure functions:**

a) `computeStreak(entryDates: string[], today: string): number` — takes a sorted-DESC array of date strings and today's date. Count consecutive days backward from today. If today has no entry, start counting from yesterday. Walk backward day-by-day using `new Date(y, m-1, d-1)` arithmetic, checking if each date exists in a `Set` built from entryDates. Return the count.

b) `getLookbackDates(date: string): Array<{ label: string; date: string }>` — compute fixed lookback dates: 30 days ago, 90 days ago, 1 year ago from the given date. Use `new Date(y, m-1, d-N)` for day offsets and `new Date(y-1, m-1, d)` for year offset. Format back to YYYY-MM-DD. Return array of `{ label, date }` where label is "30_days_ago", "90_days_ago", "1_year_ago".

Also export a helper `getLookbackLabel(today: string, entryDate: string): string` that returns the period label for a given entry date relative to today (used in API response).

**3. Create `app/api/journal/today/route.ts`:**

Single GET handler that:
- Authenticates with `supabase.auth.getUser()`, returns 401 if no user
- Reads `date` query param (default: use `getLocalDateString()` if missing, but prefer requiring it from client)
- Instantiates `JournalEntriesDB(supabase)`
- Executes 3 queries in `Promise.all`:
  - `journalDB.getEntryByDate(userId, date)` for today's entry
  - `journalDB.getRecentEntryDates(userId, date, 400)` for streak dates
  - `journalDB.getEntriesForDates(userId, getLookbackDates(date).map(d => d.date))` for On This Day
- Computes streak via `computeStreak(recentDates, date)`
- Returns JSON: `{ entry: {...} | null, streak: number, on_this_day: Array<entry + period label> }`
- The entry in the response should include id, mood, title, content, word_count (not the full timestamps)
- On This Day entries should include id, entry_date, mood, title, content (full content for dashboard teaser preview), and a `period` field (e.g., "30_days_ago")
- Follow project error handling: try/catch -> log.error -> NextResponse.json({ error }, { status: 500 })

**4. Create `app/api/journal/on-this-day/route.ts`:**

GET handler for the journal page's full On This Day view:
- Auth check, read `date` query param
- Call `getLookbackDates(date)` then `journalDB.getEntriesForDates(userId, dates)`
- Return `{ entries: Array<entry + period label> }` with full entry content
- This is a separate endpoint because the journal page needs it independently from the today widget data

Follow all project conventions: `createClient()` per request, Zod validation not needed for GET params (just query string parsing), `log` from `@/lib/logger`, proper error format.

**1. Create `app/api/journal/[id]/links/route.ts` with 3 handlers:**

a) `GET` — fetch links for entry. Auth check, get entry `id` from params. Use `JournalEntryLinksDB.getLinksForEntry(id)` to get raw links. Then enrich with names: collect link_ids by type, batch-query habits table for habit names and tasks table for task titles. Return `{ links: Array<{ id, link_type, link_id, name, created_at }> }`.

For name enrichment:
- Collect habit link_ids, query `supabase.from("habits").select("id, name").in("id", habitIds)`
- Collect task link_ids, query `supabase.from("tasks").select("id, title").in("id", taskIds)`
- Build a Map of id -> name, merge into link objects
- If a linked item was deleted, use a fallback name like "(deleted)"

b) `POST` — add a link. Auth check, parse body with `journalLinkSchema` from `lib/validations/journal.ts` via `validateRequestBody`. Verify entry ownership (query entry by id + user_id). Call `linksDB.addLink({ entry_id: id, link_type, link_id })`. Return 201 with `{ link }`.

c) `DELETE` — remove a link. Auth check, read `link_id` from query params (the link row's ID, not the target entity ID). Call `linksDB.removeLink(linkId, entryId)`. Return `{ success: true }`.

All handlers follow Next.js 16 pattern: `{ params }: { params: Promise<{ id: string }> }`, destructure with `const { id } = await params;`.
