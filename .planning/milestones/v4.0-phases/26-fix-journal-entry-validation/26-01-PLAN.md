Output: Fixed Zod schemas, updated TypeScript types, DB migration for nullable mood, updated tests.

  <action>
    **1. Fix Zod schema (`lib/validations/journal.ts`):**
    - Change `title` from `z.string().trim().min(1, "Title is required").max(200, ...)` to `z.string().trim().max(200, "Title must be 200 characters or less").default("")`. Title is never sent by autosave; it should default to empty string.
    - Change `mood` from `z.number().int().min(1).max(5).default(3)` to `z.number().int().min(1).max(5).nullable().default(null)`. The app sends `null` on mood deselect (per STATE.md design decision). `.default(null)` handles the case where mood is `undefined` (not sent at all).
    - Leave all other fields unchanged. The `journalEntryUpdateSchema` derives from `journalEntryFormSchema.partial()` so fixing the base schema automatically fixes the update schema.

    **2. Update TypeScript types (`lib/db/types.ts`):**
    - In `JournalEntry` interface: change `mood: number` to `mood: number | null`
    - In `JournalCalendarDay` interface: change `mood: number` to `mood: number | null`
    - `JournalEntryInsert` and `JournalEntryUpdate` are derived from `JournalEntry` via Omit/Partial, so they inherit the change automatically.

    **3. Clean up API route (`app/api/journal/route.ts`):**
    - In the POST handler, change `mood: validation.data.mood ?? 3` to `mood: validation.data.mood`. The Zod schema now handles the default (null). The `?? 3` fallback masked the bug and contradicts the nullable design.
    - Similarly, change `content: validation.data.content ?? { type: 'doc', content: [] }` to just `content: validation.data.content` (schema already defaults it).
    - Same for `word_count`, `tags`, `prompt_key` -- the Zod schema defaults handle all of these, so remove the `??` fallbacks. Pass `validation.data` fields directly.

    **4. Update mood-related component types:**
    - In `components/journal/journal-mood-dot.tsx`:
      - Change `moodDotColor(mood: number)` to `moodDotColor(mood: number | null)` and add early return for null: `if (mood === null) return "bg-muted-foreground";`
      - Change `JournalMoodDotProps.mood` from `number` to `number | null`
    - In `components/journal/journal-calendar.tsx`: The `entry.mood` is already conditionally rendered (`{entry && <JournalMoodDot mood={entry.mood} .../>}`). With nullable mood, this still works because `JournalMoodDot` now handles null with the muted fallback color. No structural change needed, but verify `entry.mood` passes TypeScript now that the type is `number | null`.
    - In `components/journal/journal-timeline-card.tsx`: The mood lookup `MOODS.find((m) => m.value === entry.mood)` already handles null correctly -- `find` returns undefined when mood is null, and the template already checks `{mood && <span>...}`. No code change needed, but verify TypeScript compiles cleanly.

    **5. Create DB migration (`supabase/migrations/20260223200001_make_mood_nullable.sql`):**
    ```sql
    -- Make mood column nullable to match app-level null semantics
    -- (mood deselect sends null, per design decision in STATE.md)
    ALTER TABLE journal_entries ALTER COLUMN mood DROP NOT NULL;
    ALTER TABLE journal_entries ALTER COLUMN mood SET DEFAULT NULL;
    ```
    This aligns the DB with the design decision: "Mood onChange passes null (not 0) for deselect -- cleaner API matching nullable DB column."
  </action>
